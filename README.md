# Snort-IDS-IPS-Detection-and-Evasion-Lab

## Objective
[Brief Objective]

The objective of this project is to configure and evaluate Snort as both an Intrusion Detection System (IDS) and Intrusion Prevention System (IPS) within a controlled lab.
The project aims to:
- Monitor and analyse normal network traffic (ICMP, HTTP, SSH)
- Simulate malicious activity using tools like Hydra and Nmap
- Evaluate Snort's ability to detect reconnaissance, brute force, and evasion techniques
- Correlate Snort alerts with Wireshark packet captures for validation
- Assess false positives and fine tune detection rules for improved accuracy
- Compare IDS and IPS functionality in terms of detection and response behaviour
- 
## Skills Learned

- Snort IDS/IPS configuration and rule writing
- Network traffic analysis using Wireshark
- Attack simulations (Nmap, Hydra)
- Detection of scanning, brute force, and evasion techniques
- False positive analysis and MITRE ATT&CK mapping
- SOC-style alert correlation and investigation

## Tools used

- Snort (IDS/IPS)
- Wireshark (Packet Analysis)
- Nmap (Network Scanning)
- Hydra (Brute Force Attacks)
- Kali Linux (Attack Platform)
- Metasploitable 2 (Target System)
- GitHub (Documentation and Version Control)

Nmap was used to perform a TCP SYN scan to imitate reconnaissance activity against the target system. The Nmap scan "Nmap -sS 172.30.1.28" identifies open ports and running services and it is commonly used by attackers before exploitation.

The TCP SYN Scan also known as a stealth scan sends SYN packets to the target ports and analyses them to determine whether the ports are open, closed, or filtered. The scan revealed multiple open servies on the target system including SSH (port 22) and HTTP (port 80) as the target system is intentionally configured with vulnerable services for security testing and IDS evaluation. A larger number of open ports increase the attack surface which provides multiple potential entry points for exploitation.

Based on the Nmap findings, Hydra was selected to perform dictionary based brute force attacks to imitate an attacker trying to gain unauthorized access through credentials.
Initially, we considered using SSH services for brute force simulations based on Nmaps findings. However, during testing it was clear that the attack could not proceed due to SSH cryptographic incompatibility between the attacker and the target system. The Metasploitable 2 SSH service only supports MAC algorithms such as HMAC-MD5 and HMAC-SHA1, whereas Kali SSH client uses modern cruptographic standards such as HMAC-SHA2-256 and HMAC-SHA2-512. This means that the key exchange process failed which prevented authentication attempts being made.

# CONTROLLED EVASION SIMULATION

##

We will now perform a fragmented Nmap scan using the -f option to imitate packet fragmentation evasion technique. This scan evaluates whether spliting packets into smaller fragments could reduce the effectiveness of IDS monitoring while successfully identifying open ports and servies on the target system. Attackers use packet fragmentation to obscure scan signatures and bypass IDS inspection mechanisms that might fail to properly reassemble fragment traffic before analysis. 

Fragmented Nmap scan:

<img width="560" height="540" alt="image" src="https://github.com/user-attachments/assets/2b170273-dab2-47d4-a6a0-034155fffdc2" />

The Nmap scan shows that it has successfully identified multiple exposed ports such as FTP, SSH, HTTP, and more despite using packet fragmentation. This shows that the fragmentation did not prevent services from being discovered.

Wireshark was used to confirm that the fragmented Nmap scan produced multiple fragmented IP packets during transmission. 

<img width="1632" height="292" alt="image" src="https://github.com/user-attachments/assets/dc51c4df-e7fa-4d7e-bee0-322505cffe1e" />

Wireshark packet capture exhibited "Fragment IP Protocol" and "Resembled in" entries which demonstrates that the scan traffic had been divided into smaller fragments and was later reconstructed. This shows that packet fragmentation can obscure scanning activity. It can also reduce IDS visibility at the packet level.

Snort did not generate any intrusion detection alerts during the fragmented Nmap scan. This shows that the fragmented packet structure may have reduced the visibility of the scan activity to the IDS. This prevented signature based detection from being triggered. This does not confirm full evasion but it indicates reduced detection capability under fragmented traffic conditions.

# NETWORK I9NTRUSION ANALYSIS AND CLASSIFICATION 

ICMP/Ping traffic

ICMP (ping) traffic was observed in Wireshark as an echo requestand reply packets. As Snort did not generate any meaningful alerts, it indicates that the traffic was treated as normal under default rule settings. This means that the activity was classified as a true negative as no malicious behaviour was present and no detection was expected.

SSH Login Activity

A remote SSH login attept was performed to mimick legitimate administrative access. This was confirmed by Wireshark presenting the TCP three way handshake on port 22 followed by the SSH protocol and encrypted session establishment.

Nmap SYN Scan

Nmap was used to perform as TCP SYN scan to imitate reconnaissance activity against the target system. Wireshark captured multiple SYN packets sent across a range of ports which confirmed systematic port enmureation.
Snort successfully detected this activity and generated alerts associated with reconnaissance and service probing activity. this included classifications such as attempted information leak. Therefore, this activity is classified as a true positive as malicious scanning behaviour was identified correctly.

Slow/Stealth Scan

A stealth SYN scan was executed to imitate reconnaissance activity an Wireshark was used to confirm that the SYN packets were distributed across multiple ports. 
Snort did not generate any alerts during this activity as it was not detected by signature based rules. This is due to ite reduced rate and fragmented timing therefore, it was classified as a false negative. This is as malicious activity did occur but it was not detected.

HTTP and URL encoded traffic

Curl was used to generate standard HTTP and URL encoded requests to imitate normal user behaviour. Wireshark was used to confirm successful HTTP GET requests and encoded payloads which were correctly decoded by the web server before it was processed. This resulted in the server responding with 404 Not Found and 301 Moved Permanently showing normal application behaviour. 

Snort generated alerts despite the traffic being non malicious and legitimate. The traffic was classified as Potentially Bad Traffic even though there was no evidence of exploitation, payload delivery or abnormal protocol behaviour observed within Wireshark. This means the results were classified as false positives demonstrating that Snort's signature based detection may flag normal traffic when using low confidence rules.

Hydra SSH Brute Force Activity

Hydra was used to imitate attempts to brute force against SSH with the RockYou password list against the target system. Wireshark confirmed multiple TCP connection attempts between the attacker and the target system showing repeated SYN and SYN-ACK exchanged on port 22. However,no completed SSH sessions were established and no payload level authentication data was observed. This is due to encryption negociation failure. 

Snort did not generate any alerts as IDS did not classify the repeated connection attempts as malicious under the current rule set. This is a limitaion in detection capability as low level and incomplete authentication attempts did not trigger signature based alerts without specific brute force detection rules.

Fragmented Nmap Scan

To imitate IDS evasion we executed a fragmented Nmap scan using packet fragmentation techniques. Wireshark captured the packets and confirmed IP fragments and reassembly processes. Snort did not generate any alerts as the scan reduced detection effectiveness when the traffic is fragmented at the packet level. Therefore it was classified as a false negative as reconnaissance activity occured but it was not detected by IDS.

Overall Snort is effective at detecting and generating alerts against clear reconnaissance and bruteforce patterns but miss low and slow or fragmented attacks. Wireshark confirmed all classified events were present at packet level which allowed accurate validation of IDS behaviour.
